# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

**Agentic RAG Document Intelligence API** — a production-shaped RAG REST service. Users upload PDFs / DOCX / TXT → chunks are embedded and stored in ChromaDB → `/ask` and `/ask/stream` endpoints return cited answers generated by Groq's hosted Llama 3.3 70B, with conversational memory per session. `/evaluate` and a CLI script score retrieval + generation quality via RAGAS.

**Stack is 100% free** — only external signup is a free `GROQ_API_KEY`. Embeddings run locally via `BAAI/bge-small-en-v1.5` (sentence-transformers). No paid APIs anywhere.

## Architecture

This is a RAG (Retrieval-Augmented Generation) system. The core data flow has two paths:

**Ingestion path** (upload → background task):
`POST /upload` → validates file → registers doc in `DocumentRegistry` → fires `BackgroundTask(_ingest_document)` → `extract_text()` → `chunk_documents()` → `VectorStoreManager.store_chunks()` (Chroma collection `documents`, per-chunk metadata `{doc_id, filename, page_number, file_type}`) → `DocumentRegistry.mark_ready()`.

**Query path** (ask → LLM answer):
`POST /ask` (or `POST /ask/stream`) → checks `DocumentRegistry` for doc status → `get_answer()` / `astream_answer()` → `_ManagerRetriever` pulls conversation history from `SessionMemory` → LCEL chain (`create_history_aware_retriever` → `create_retrieval_chain`) retrieves chunks from Chroma → `ChatGroq(llama-3.3-70b-versatile)` generates answer with citations → `SessionMemory.add_turn()` → returns `AnswerResponse` with `sources` (or streams SSE events).

**Evaluation path**:
`POST /evaluate` or `scripts/evaluate.py` → `services.evaluation.score_single/score_batch` → RAGAS with `LangchainLLMWrapper(ChatGroq)` + `LangchainEmbeddingsWrapper(HuggingFaceEmbeddings)` → returns faithfulness, answer_relevancy, and (if ground_truth given) context_precision.

### Key design decisions

**Lazy initialization everywhere** — `get_settings()` is `@lru_cache`'d and never called at module import time. `VectorStoreManager._ensure_initialized()` runs on first use. The app starts successfully even with missing env vars; errors surface at request time.

**Single Chroma collection with metadata filter** — Chroma has no Pinecone-style namespaces. Every chunk carries `doc_id` in its metadata, and per-document queries pass `filter={"doc_id": doc_id}`. Global queries omit the filter. `delete_document(doc_id)` uses `collection.delete(where={"doc_id": doc_id})`. This scales better than a collection-per-doc approach and keeps the "query across all docs" case trivial.

**Local embeddings via BGE** — `BAAI/bge-small-en-v1.5` (384-dim) runs on CPU via `sentence-transformers`. First load downloads ~130 MB; the Dockerfile pre-downloads it at build time so the container starts fast. `encode_kwargs={"normalize_embeddings": True}` because BGE expects normalized vectors for cosine similarity.

**`split_documents()` not `split_text()`** — The document processor uses LangChain's `split_documents()` so every chunk inherits its parent's metadata (`doc_id`, `filename`, `page_number`, `file_type`). This metadata is what populates `SourceChunk` in responses.

**LCEL streaming chain, not `ConversationalRetrievalChain`** — The deprecated legacy chain doesn't stream cleanly. We compose with `create_history_aware_retriever` + `create_retrieval_chain` so `.astream_events()` yields per-token deltas for SSE.

**SSE over `StreamingResponse`** — `/ask/stream` returns a `StreamingResponse(..., media_type="text/event-stream")`. Each event is formatted as `data: {json}\n\n`. Event types: `token` (streaming delta), `sources` (final citations, once), `done` (terminator), `error` (on failure).

**In-memory state** — `DocumentRegistry` and `SessionMemory` are module-level singletons backed by dicts with `threading.Lock`. They reset on server restart. The registry tracks document status (`processing` → `ready` / `failed`); the session memory keeps the last `MAX_CONVERSATION_TURNS` (default 5) Q&A pairs per `session_id`. These are v1 — Redis/Postgres is the v2 path.

### LangChain import paths (v1.x)

This project uses LangChain 1.x where several modules moved packages:

- `RecursiveCharacterTextSplitter` → `from langchain_text_splitters import ...`
- `Document` → `from langchain_core.documents import ...`
- `BaseRetriever` → `from langchain_core.retrievers import ...`
- `ChatGroq` → `from langchain_groq import ChatGroq`
- `HuggingFaceEmbeddings` → `from langchain_huggingface import HuggingFaceEmbeddings`
- `Chroma` → `from langchain_chroma import Chroma`
- `create_history_aware_retriever`, `create_retrieval_chain` → `from langchain.chains import ...`
- Document loaders (`PyPDFLoader`, `Docx2txtLoader`, `TextLoader`) → `from langchain_community.document_loaders import ...`

Do **not** use the old `langchain.text_splitter` path or `langchain_classic.chains.ConversationalRetrievalChain`.

### Configuration

All config is in `core/config.py` via `Settings(BaseSettings)`. Key tunables:

- `GROQ_API_KEY` — required, free from console.groq.com
- `GROQ_CHAT_MODEL` — defaults to `llama-3.3-70b-versatile`
- `LLM_TEMPERATURE` — defaults to `0.0`
- `LLM_MAX_TOKENS` — defaults to `1024`
- `EMBEDDING_MODEL` — defaults to `BAAI/bge-small-en-v1.5` (384 dims)
- `EMBEDDING_DEVICE` — defaults to `cpu` (set to `cuda` on GPU hosts)
- `CHROMA_PERSIST_DIR` — defaults to `./data/chroma`
- `CHROMA_COLLECTION` — defaults to `documents`
- `UPLOAD_DIR` — defaults to `./data/uploads`
- `MAX_UPLOAD_SIZE_MB` — defaults to `25`
- `ALLOWED_EXTENSIONS` — `{.pdf, .txt, .docx, .md}`
- `RETRIEVAL_K` — chunks retrieved per query (default 5)
- `CHUNK_SIZE` / `CHUNK_OVERLAP` — splitter params (default 1000/200)
- `MAX_CONVERSATION_TURNS` — session history depth (default 5)
- `CORS_ORIGINS` — defaults to `["http://localhost:8501"]`

The Chroma persist directory is auto-created on first use.

### Testing

Tests use `pytest` with `httpx`/`TestClient`. `pytest.ini` sets `pythonpath = .` so `core/`, `api/`, and `services/` are importable without a package install. `conftest.py` sets dummy env vars (`GROQ_API_KEY="test"`, etc.) so `Settings` validation passes without a real `.env`, and provides three fixtures:

- `mock_vector_store` — replaces `vector_store_manager` with a `MagicMock` in both `services.vector_store` and `api.routes`.
- `mock_qa` — patches both `get_answer` and `astream_answer` in `api.routes` so tests don't hit Groq.
- `client` — `TestClient(app)` that depends on both mocks above.

The `_reset_registries` fixture is `autouse=True` and clears `document_registry._store` and `session_memory._sessions` between every test.

## Git Workflow

**Remote:** `https://github.com/saahilnaik9/rag-document-intelligence-api`

After completing each PLAN.md step (or any meaningful chunk of work), commit and push:

```bash
# Stage specific files — never use git add -A (avoids accidentally committing .env or data/)
git add <file1> <file2> ...

# Commit — message format: <type>: <short present-tense description>
# Types: feat | fix | refactor | test | docs | chore | style
# Examples:
#   feat: add /upload endpoint with background ingestion
#   feat: implement LCEL streaming chain for /ask/stream
#   test: add session memory eviction tests
#   chore: add requirements.txt and project scaffolding
git commit -m "type: description"

# Push
git push origin main
```

**Commit message rules:**

- Present tense, imperative mood ("add" not "added")
- Under 72 characters for the subject line
- No period at the end
- Reference the step completed, e.g. `feat: scaffold core config and health endpoint (step 1)`
- Never include "Claude", "AI-generated", or task metadata in the message
- Never skip hooks with `--no-verify`

**What never goes in a commit:**

- `.env` (contains `GROQ_API_KEY`) — gitignored
- `data/` (uploads + Chroma vectors) — gitignored
- `venv/` — gitignored
- `__pycache__/`, `*.pyc` — gitignored
- `eval_results/` — gitignored

## What NOT to change without asking

- **LLM provider**: switching from Groq to Claude/GPT/etc. requires replacing `ChatGroq` with a different LangChain chat model. The architecture is designed for this swap — it's a one-line change plus env var rename — but confirm before doing it.
- **Chroma → Pinecone**: the `VectorStoreManager` abstracts the store, but the metadata-filter semantics vs. namespaces differ. Confirm before switching.
- **In-memory registries → Redis**: documented as v2 work; don't do it piecemeal.
- **LangChain version**: pinned to 1.x. Don't upgrade/downgrade without checking import-path compatibility.
