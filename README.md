# Agentic RAG Document Intelligence API

A production-shaped **Retrieval-Augmented Generation (RAG)** service. Upload PDF, DOCX, or TXT files → the system embeds and indexes them → ask questions in natural language → get cited answers generated by a large language model, with real-time streaming and conversational memory.

**100% free to run.** The only external signup required is a free [Groq API key](https://console.groq.com). All embeddings run locally on your CPU.

---

## What It Does

1. **Upload documents** — PDF, DOCX, TXT, or Markdown files up to 25 MB
2. **Auto-indexing** — Documents are chunked, embedded locally, and stored in ChromaDB (runs in the background while you wait)
3. **Ask questions** — Natural language questions return answers with `[filename:page]` citations pulled directly from your documents
4. **Streaming answers** — Tokens stream to the browser in real-time via Server-Sent Events
5. **Conversation memory** — Each session remembers the last 5 exchanges so you can ask follow-up questions
6. **Quality evaluation** — Built-in RAGAS scoring (faithfulness, answer relevancy, context precision)

## Architecture

```
┌──────────────────────────────────────────────────────────────────┐
│                        Next.js Frontend                          │
│   Sidebar (upload/select)  ←→  ChatWindow (streaming)           │
└───────────────────────────┬──────────────────────────────────────┘
                            │ HTTP / SSE
┌───────────────────────────▼──────────────────────────────────────┐
│                     FastAPI Backend                              │
│                                                                  │
│  POST /upload  →  BackgroundTask                                 │
│       │                  │                                       │
│       │           extract_text()                                 │
│       │           chunk_documents()      ┌─────────────────┐    │
│       │           store_chunks()  ──────►│   ChromaDB       │    │
│       │                                  │  (local vectors) │    │
│  POST /ask/stream                        └────────┬────────┘    │
│       │                                           │              │
│       │  SessionMemory (last 5 turns)             │              │
│       │  ↓                                        │              │
│       │  condense follow-up question              │              │
│       │  retrieve top-K chunks  ◄─────────────────┘              │
│       │  ↓                                                       │
│       │  ChatGroq (llama-3.3-70b-versatile)                      │
│       │  ↓                                                       │
│       └─ stream tokens + sources back via SSE                   │
└──────────────────────────────────────────────────────────────────┘
```

### Data Flow

**Ingestion (upload → background)**

```
POST /upload → validate → register doc → background task starts
  → extract_text()       — PyPDF / Docx2txt / TextLoader
  → chunk_documents()    — RecursiveCharacterTextSplitter (1000 chars / 200 overlap)
  → store_chunks()       — BGE embeddings (384-dim) → ChromaDB
  → mark_ready()
```

**Query (ask → answer)**

```
POST /ask/stream → check doc status → load session history
  → condense question (if follow-up) via Groq
  → retrieve top-5 chunks from ChromaDB (filtered by doc_id if scoped)
  → build context string with [filename:page] markers
  → stream answer from ChatGroq (llama-3.3-70b-versatile)
  → emit: token events → sources event → done event
  → save turn to SessionMemory
```

## Quick Start (Local Development)

### Prerequisites

- Python 3.11
- Node.js 20+
- A free Groq API key from [console.groq.com](https://console.groq.com)

### 1 — Clone and set up environment

```bash
git clone https://github.com/saahilnaik9/rag-document-intelligence-api
cd rag-document-intelligence-api
```

### 2 — Backend setup

```bash
# Create virtual environment
py -3.11 -m venv venv

# Install dependencies
venv\Scripts\pip install -r requirements.txt

# Create .env and add your Groq key
copy .env.example .env
# Open .env and set GROQ_API_KEY=your_key_here
```

### 3 — Frontend setup

```bash
cd frontend
npm install
cd ..
```

### 4 — Run both servers

**Terminal 1 — Backend:**
```bash
venv\Scripts\uvicorn main:app --reload
# API running at http://localhost:8000
# Swagger UI at http://localhost:8000/docs
```

**Terminal 2 — Frontend:**
```bash
cd frontend
npm run dev
# App running at http://localhost:3000
```

Open `http://localhost:3000`, upload a PDF, wait for it to show "ready", then start asking questions.

> **Note:** The first upload triggers a ~130 MB download of the BGE embedding model. Subsequent uploads are instant.

## API Reference

All endpoints are also available interactively at `http://localhost:8000/docs`.

### Upload a document
```http
POST /upload
Content-Type: multipart/form-data

file: <your PDF, DOCX, TXT, or MD file>
```
Returns `{ doc_id, filename, status: "processing" }` immediately. Processing continues in the background.

### Check document status
```http
GET /documents/{doc_id}
```
Returns `{ doc_id, filename, status, chunk_count }`. Poll until `status` is `"ready"`.

### List all documents
```http
GET /documents
```

### Delete a document
```http
DELETE /documents/{doc_id}
```
Removes from both the registry and the vector store.

### Ask a question (streaming)
```http
POST /ask/stream
Content-Type: application/json

{
  "question": "What are the main findings?",
  "session_id": "<uuid>",
  "doc_id": "<uuid or omit to search all docs>"
}
```
Returns a `text/event-stream`. Events:
- `{"type": "token", "data": "..."}` — streamed answer tokens
- `{"type": "sources", "data": [...]}` — cited chunks (once, at the end)
- `{"type": "done"}` — stream complete
- `{"type": "error", "data": "..."}` — on failure

### Ask a question (non-streaming)
```http
POST /ask
```
Same request body, returns `{ answer, sources, session_id }` when complete.

### Evaluate answer quality
```http
POST /evaluate
Content-Type: application/json

{
  "question": "...",
  "answer": "...",
  "contexts": ["chunk1", "chunk2"],
  "ground_truth": "..." // optional — enables context_precision
}
```
Returns `{ faithfulness, answer_relevancy, context_precision }`.

### Health check
```http
GET /health
```
Returns `{ "status": "ok" }`.

## Evaluation

The project uses [RAGAS](https://docs.ragas.io) to measure RAG quality along three axes:

| Metric | What it measures |
|---|---|
| **Faithfulness** | Is the answer grounded in the retrieved context? (0–1, higher = less hallucination) |
| **Answer Relevancy** | Does the answer actually address the question? (0–1) |
| **Context Precision** | Are the retrieved chunks relevant to the question? (requires ground truth) |

### Run the batch evaluator

```bash
venv\Scripts\python scripts/evaluate.py eval_dataset/sample_qa.jsonl
```

This runs the full RAG pipeline for every Q&A pair in the JSONL file, scores each one with RAGAS, prints a table, and writes results to `eval_results/{timestamp}.json`.

To scope retrieval to a specific document:
```bash
venv\Scripts\python scripts/evaluate.py eval_dataset/sample_qa.jsonl --doc-id <uuid>
```

The JSONL format:
```jsonl
{"question": "What is X?", "ground_truth": "X is ..."}
```

## Configuration

All settings live in `core/config.py` and are read from `.env`.

| Variable | Default | Description |
|---|---|---|
| `GROQ_API_KEY` | **required** | Free from console.groq.com |
| `GROQ_CHAT_MODEL` | `llama-3.3-70b-versatile` | Groq model to use |
| `LLM_TEMPERATURE` | `0.0` | Lower = more factual |
| `LLM_MAX_TOKENS` | `1024` | Max answer length |
| `EMBEDDING_MODEL` | `BAAI/bge-small-en-v1.5` | Local embedding model |
| `EMBEDDING_DEVICE` | `cpu` | Set to `cuda` on GPU machines |
| `CHROMA_PERSIST_DIR` | `./data/chroma` | Where vectors are stored |
| `UPLOAD_DIR` | `./data/uploads` | Where uploaded files are saved |
| `MAX_UPLOAD_SIZE_MB` | `25` | Per-file size limit |
| `RETRIEVAL_K` | `5` | Chunks retrieved per query |
| `CHUNK_SIZE` | `1000` | Characters per chunk |
| `CHUNK_OVERLAP` | `200` | Overlap between chunks |
| `MAX_CONVERSATION_TURNS` | `5` | Session history depth |

## Key Design Decisions

**Why local embeddings?** `BAAI/bge-small-en-v1.5` runs on CPU with no API cost. The 384-dimension vectors are compact and fast. The first run downloads ~130 MB; Docker pre-downloads it at build time.

**Why a single Chroma collection with metadata filters instead of one collection per document?** Querying across all documents is trivial (no filter). Querying a single document adds `filter={"doc_id": ...}`. Deleting a document uses `collection.delete(where={"doc_id": ...})`. One collection scales cleanly.

**Why LCEL streaming instead of `ConversationalRetrievalChain`?** The legacy chain doesn't stream tokens cleanly. The LCEL composition (`create_history_aware_retriever` + `create_retrieval_chain`) streams per-token deltas that map directly to SSE events.

**Why in-memory registries?** Simplicity for v1. They reset on server restart. Redis/Postgres is the documented v2 path when persistence across restarts is needed.
